# ADR-0001 架构设计原则与工程实践

## 1. 状态与背景（Context）

本项目目标是在公司内部构建一套可控、可扩展、可审计的 Agent 框架，支持多模型、多数据源（当前重点为 Oracle 与 Snowflake）及多接入层（API/UI/批处理）。

约束与前提如下：

- 工程约束：面向生产环境，优先保障高可用、高维护性与可观测性。
- 业务约束：需要支持复杂查询、工具编排、权限控制、审计追踪。

## 2. 决策（Decision）

采用“分层 + 插件化 + 契约优先”的架构模型，目录层级与职责如下：

- `core/`：领域契约与核心模型（LLM、Tool、Storage、Workflow 抽象）。
- `engine/`：对话编排器（主循环、工具调用协调、错误恢复策略）。
- `capabilities/`：能力接口（数据源、文件系统、记忆等，与工具解耦）。
- `integrations/`：外部系统适配（Oracle、Snowflake、OpenAI 等）。
- `tools/`：可被模型调用的工具实现（遵循单一职责）。
- `api/`：FastAPI/Flask 接入层，负责协议转换、鉴权与错误映射。
- `contracts/`（后续补充）：API 请求响应与 UI 事件模型统一定义。
- `config/`：配置模型与加载逻辑（环境区分、密钥引用、参数校验）。
- `observability/`（后续补充）：日志、指标、链路追踪、审计记录。
- `tests/`：单元、契约、集成、端到端测试分层。
- `docs/`：ADR、架构说明、运行手册与合规记录。

## 3. 架构原则（Architecture Principles）

### 3.1 分层边界清晰

- 上层只依赖下层抽象，不依赖具体实现。
- `engine` 不直接依赖具体数据库驱动，仅依赖 `capabilities` 抽象。
- `tools` 不内嵌数据库连接逻辑，只调用能力接口。

### 3.2 契约优先（Contract-First）

- 对外输入输出全部通过类型模型定义（Pydantic 或 dataclass）。
- LLM 请求、工具参数、查询结果、API 协议都有独立模型。
- 所有跨层交互通过明确契约，避免隐式字段传递。

### 3.3 插件化扩展

- 新增数据源通过 `DataSource` 接口扩展，不修改编排器核心逻辑。
- 新增工具通过 `ToolRegistry` 注册，不破坏既有工具。
- 新增模型供应商通过 `LlmService` 适配器扩展。

### 3.4 安全默认（Secure by Default）

- 数据库默认只读、白名单 schema、SQL 类型限制（优先只允许 SELECT）。
- 密钥不落库、不写代码，统一由环境变量或密钥服务注入。
- API 层必须有鉴权和授权，核心链路全量审计。

### 3.5 可测试优先

- 每个抽象接口都应有 mock/stub 实现，支持离线测试。
- 单测覆盖核心决策逻辑，契约测试覆盖适配器行为，集成测试覆盖关键链路。

## 4. 高可用设计（Availability）

### 4.1 无状态服务与多实例部署

- API 与编排器进程保持无状态，状态存放在外部存储。
- 支持容器化部署与水平扩展（多副本 + 负载均衡）。

### 4.2 故障隔离与降级

- 外部依赖（LLM、Oracle、Snowflake）调用设置超时、重试与熔断。
- 单个数据源故障不影响其他数据源可用性。
- 当工具或数据源不可用时，提供明确降级响应和错误分类。

### 4.3 幂等与重入

- 对关键调用引入 `request_id` / `trace_id`，支持重试与幂等校验。
- 工具执行需要保证“失败可重试、成功可追踪”。

### 4.4 资源保护

- 限制 SQL 查询超时、返回行数、并发数，避免数据库雪崩。
- 对 LLM 调用设置速率限制和预算上限。

## 5. 高维护性设计（Maintainability）

### 5.1 单一职责与低耦合

- 一个模块只承担一个明确职责。
- 避免“上帝类”；编排逻辑、存储逻辑、数据源逻辑、协议逻辑分离。

### 5.2 稳定接口与版本化

- 对外 API 及核心契约模型需要版本化（例如 `/v1`）。
- 模型字段变更需兼容旧版本，重大变更走 ADR 与迁移方案。

### 5.3 可读性与一致性

- 统一命名规范、目录规范、错误码规范、日志字段规范。
- Python 包导入统一使用绝对导入路径。

### 5.4 文档即资产

- 每个核心设计决策都用 ADR 记录“背景-决策-影响”。
- 新增能力必须补充设计说明、运行手册与测试说明。

## 6. 可观测性与运维（Observability & Ops）

### 6.1 结构化日志

- 关键字段标准化：`trace_id`、`conversation_id`、`user_id`、`tool_name`、`source_id`、`latency_ms`、`status`。
- 错误日志必须包含错误分类、重试次数和上下文摘要。

### 6.2 指标体系

- 核心指标：请求成功率、P95/P99 延迟、工具成功率、数据源错误率、重试率、超时率。
- 业务指标：对话完成率、查询命中率、平均响应时长。

### 6.3 链路追踪

- 统一 trace 贯穿 API -> Engine -> Tool -> DataSource -> LLM。
- 支持问题快速定位与容量分析。

### 6.4 审计合规

- 审计记录至少包括：谁、何时、调用了什么、访问了哪个数据源、执行了什么 SQL（可脱敏存储）。
- 满足内部审计与安全稽核要求。

## 7. 数据源接入标准（Oracle / Snowflake）

为保证一致性，所有数据源适配器必须实现以下能力：

- `test_connection`：连通性检查与依赖校验。
- `execute_query`：统一查询入口，返回统一结果模型。
- `list_tables`：元数据发现能力。
- 统一异常映射：连接失败、鉴权失败、超时、SQL 语法错误、权限错误。

接入标准要求：

- 默认只读角色，最小权限原则。
- 禁止高风险语句（DDL/DML）或需显式白名单审批。
- 所有查询必须携带来源上下文用于审计。

## 8. 测试策略（Testing Strategy）

- `unit`：核心业务逻辑、路由、参数校验、错误映射。
- `contract`：适配器对抽象接口的行为一致性。
- `integration`：与真实/仿真 Oracle、Snowflake、LLM 的联调。
- `e2e`：完整对话链路与关键业务场景回归。

发布门禁建议：

- 单测通过率 100%（阻断级）。
- 核心路径最小覆盖率（建议 80%+）。
- 关键集成用例通过（数据源查询、工具编排、异常回退）。

## 9. 配置与发布策略（Config & Release）

- 配置分环境：`dev`、`test`、`staging`、`prod`。
- 密钥统一由环境变量或 Secret Manager 管理。
- 发布采用灰度/分批策略，支持快速回滚。
- 任何高风险配置（如写权限、超大并发）必须显式开启。

## 10. 影响评估（Consequences）

正向影响：

- 模块边界清晰，易于扩展新模型和新数据源。
- 故障隔离与观测能力完善，提升生产可用性。
- 规范化测试与文档体系降低维护成本。

成本与权衡：

- 初期工程投入较高（抽象、文档、测试体系）。
- 接入速度略慢于“直接写死”方案，但中长期收益更高。

## 11. 落地路线图（Roadmap）

### Phase 1（当前）

- 核心抽象、最小编排器、`query_data` 工具、Oracle/Snowflake 占位适配器、基础单测。

### Phase 2

- 真实 Oracle/Snowflake 驱动接入、只读 SQL 策略、错误映射、连接池与超时控制。

### Phase 3

- API 鉴权授权、审计日志、指标与追踪、契约测试与集成测试补齐。

### Phase 4

- 灰度发布、SLO/SLA 定义、容量评估、运维手册与故障预案完善。

## 12. 决策结论（Status）

**Accepted**（生效）。  
后续若架构边界发生变化，需新增 ADR 记录并关联本文件进行版本化演进。
